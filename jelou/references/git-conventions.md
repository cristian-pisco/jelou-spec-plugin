# Git Workflow Conventions

> This document defines the Git workflow enforced by the Jelou Spec Plugin. All git operations are performed by the git-agent (Haiku tier) under orchestrator direction. The git-agent never makes judgment calls — it executes predetermined operations.

## Branch Naming

All task branches follow a single naming convention:

```
spec/<task-slug>
```

Examples:
- `spec/add-user-auth`
- `spec/payment-webhook-retry`
- `spec/dashboard-analytics`

### Rules

- One logical branch name per task, used consistently across **all** affected service repos (Decision #40).
- The branch name is derived from the task slug, which is set during `/jlu:new-task`.
- Branch names are lowercase, hyphen-separated, and descriptive.

## Worktree Management

Each service repo participating in a task gets a dedicated worktree:

```
<repo-root>/.worktrees/<task-slug>
```

### Lifecycle

| Event | Action |
|-------|--------|
| `/jlu:new-task` | Creates worktrees in all confirmed affected repos |
| During execution | All code changes happen in the worktree, not the main working tree |
| `/jlu:report-task` | Shows stale worktrees (tasks in `done` or `closed` state) with cleanup prompt |
| `/jlu:close-task` | Worktree can be cleaned up (not automatic) |

### Stale Detection (Decision #17)

No automatic worktree cleanup. `/jlu:report-task` identifies stale worktrees — those associated with tasks in `done` or `closed` state — and prompts the user to clean them up. Worktrees persist until explicitly removed.

## Commit Conventions

### Priority Order

1. **Project convention**: If the repo has an established commit convention (detected from git history, `.commitlintrc`, or `CONVENTIONS.md`), use it.
2. **Conventional commits fallback**: If no project convention exists, use [Conventional Commits](https://www.conventionalcommits.org/):
   ```
   <type>(<scope>): <description>

   [optional body]
   ```
   Types: `feat`, `fix`, `test`, `refactor`, `docs`, `chore`, `ci`

### Rules

- Commit messages are generated by the git-agent based on the changes made.
- Each commit should represent a logical unit of work (e.g., one phase completion, one test suite, one refactor pass).
- The git-agent never amends or force-pushes. New commits only.

## Protected Branch Restrictions

The git-agent is **strictly forbidden** from pushing to:

- `main`
- `master`
- `alpha`

Any attempt to push to a protected branch is blocked and escalated to the user. This restriction is absolute and cannot be overridden by agents.

## Unrelated Change Detection

If the worktree contains unexpected or unrelated changes (files modified that are not part of the task scope):

1. The git-agent **blocks** the operation.
2. The orchestrator is notified.
3. The user is presented with the unrelated changes and asked how to proceed:
   - Stash the changes.
   - Include them in the commit (with explicit approval).
   - Abort and let the user handle it manually.

## PR Strategy (Decision #40)

### Single PR per Repo

Each affected service repo gets exactly one PR for the task:

- **Branch**: `spec/<task-slug>`
- **Target**: The repo's default branch (typically `main` or `develop`)
- **Title**: Derived from the task title
- **Description**: Generated from SPEC.md + PROPOSAL.md + TASKS.md

### Multi-Service Coordination

For multi-service tasks, PRs are coordinated across repos:

- All PRs reference the same task slug.
- PR descriptions cross-reference related PRs in other repos.
- Merge order follows the dependency order defined in PROPOSAL.md.
- The orchestrator tracks PR status across all repos in TASKS.md.

## Git Operations Summary

| Operation | Who Decides | Who Executes |
|-----------|-------------|-------------|
| Branch creation | Orchestrator (via `/jlu:new-task`) | git-agent |
| Worktree creation | Orchestrator (via `/jlu:new-task`) | git-agent |
| Staging + committing | Orchestrator (after phase completion) | git-agent |
| Pushing | Orchestrator (at appropriate checkpoints) | git-agent |
| PR creation | Orchestrator (at `ready_to_publish`) | git-agent |
| Branch protection | Hardcoded rule | git-agent enforces |
